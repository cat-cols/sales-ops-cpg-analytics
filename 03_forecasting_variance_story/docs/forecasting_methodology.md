# Forecasting Methodology

## Objective

Project 3 simulates a business planning workflow that compares weekly forecasted sales against actual sales.

The purpose is not only to forecast sales, but to evaluate forecast accuracy, identify variance drivers, and support business recommendations.

---

## Forecast grain

The forecast is modeled at the weekly store/SKU/channel grain:

```text
week_start_date + store_code + sku + channel + plan_version
````

This grain supports detailed diagnostics while allowing rollups to:

* total business
* week
* channel
* region
* store
* SKU
* product family
* business event

---

## Forecast horizon

The synthetic dataset covers:

```text
2025-01-06 through 2026-03-30
```

The model uses weekly periods beginning on Monday.

---

## Input data

The Project 3 source extract is generated by:

```text
03_forecasting_variance_story/scripts/generate_project3_forecast_inputs.py
```

The generator uses shared seed files:

```text
shared/seeds/location_seed.csv
shared/seeds/product_seed.csv
shared/seeds/channel_seed.csv
shared/seeds/calendar_seed.csv
```

The output source extract is:

```text
03_forecasting_variance_story/data/source_extracts/forecast_actuals_weekly.csv
```

---

## Shared seed strategy

Shared seed files represent reusable reference data.

Project-specific source extracts remain inside the project folder.

This creates a realistic pattern:

```text
shared reference data → project-specific source extract → raw/staging/int/mart model
```

This mirrors how a real analytics environment often uses common master data across multiple reporting workflows.

---

## Forecast construction

The simulated forecast uses the following components:

```text
base product demand
location/store multiplier
channel unit multiplier
seasonality multiplier
random forecast noise
```

Forecast units are calculated conceptually as:

```text
base_units
× store_multiplier
× channel_unit_multiplier
× seasonality_multiplier
× forecast_noise
```

Forecast net sales are calculated as:

```text
forecast_units × forecast_unit_price
```

---

## Actual sales construction

Actual units begin with forecast units and are then adjusted for:

```text
actual demand noise
promo effects
stockout effects
channel-level forecast bias
```

Actual net sales are calculated as:

```text
actual_units × actual_unit_price
```

---

## Seasonality

The simulator includes simple seasonal patterns:

```text
summer_lift
holiday_lift
winter_softness
baseline
```

These patterns create more realistic weekly sales movement.

---

## Promo effects

The simulator includes a summer retail promo event for selected SKUs.

Expected behavior:

```text
actual units increase
actual unit price decreases
net sales may increase if volume lift offsets discounting
```

This supports analysis of promo-related forecast variance.

---

## Stockout effects

The simulator includes holiday stockout pressure for selected SKU/location combinations.

Expected behavior:

```text
actual units decrease
actual sales decrease
variance is primarily volume-driven
```

This supports analysis of missed sales due to product availability issues.

---

## Partial actuals

The most recent week intentionally includes missing actuals to simulate reporting lag.

These rows are flagged using:

```text
is_partial_actual
```

Partial actuals should be excluded from finalized forecast accuracy and variance bridge reporting.

---

## Forecast accuracy metrics

The project uses:

```text
forecast bias
absolute sales error
WAPE
```

WAPE is emphasized because it weights error by actual sales volume.

This is more useful for business reporting than simple average percentage error when sales volumes vary across SKUs, stores, or channels.
